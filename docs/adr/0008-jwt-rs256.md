# ADR 0008: JWT com assinatura assimétrica RS256

- **Status:** Accepted
- **Data:** 2026-05-24
- **Deciders:** Danilo Fernando

## Contexto e problema

O requisito 5.2 exige autenticação baseada em **JWT** com Spring
Security, perfis de acesso (`USUARIO`, `DONO_RESTAURANTE`) e
extração do `clienteId` a partir do `subject` do token.

Há duas famílias principais de algoritmos de assinatura JWT:

- **HMAC simétrico (HS256/384/512):** uma única chave secreta que
  serve para assinar **e** validar. Quem tem a chave pode emitir
  tokens.
- **RSA assimétrico (RS256/384/512):** par de chaves; a chave
  **privada** assina (só o auth-service), a chave **pública** valida
  (distribuída para os consumidores).

Em um cenário multi-serviço, com HS256 cada serviço-validador
precisaria conhecer a chave secreta — qualquer um deles
comprometido daria a um atacante a capacidade de **emitir** tokens
válidos, não só de verificá-los.

## Decisão

Usar **RS256** para assinatura dos JWTs:

- A **chave privada RSA** vive apenas no `usuario-autenticacao`
  (`classpath:keys/privateKey.pem`), usada para assinar via
  `JwtTokenProvider`.
- A **chave pública RSA** é distribuída no classpath de cada
  serviço-validador (`restaurante-pedido`, `pagamento-service`,
  `restaurante-service`) em `classpath:keys/publicKey.pem`.
- Cada validador configura um `NimbusJwtDecoder.withPublicKey(...)`
  no seu `SecurityConfig`.

Claims emitidas no token:

```json
{
  "iss": "fiap-restaurante",
  "iat": 1716...,
  "exp": 1716...,
  "sub": "<UUID do usuario>",
  "email": "usuario@fiap.com",
  "nome": "Nome do Usuario",
  "groups": ["USUARIO"]
}
```

A claim `groups` é mapeada para `GrantedAuthority` sem prefixo,
permitindo `@PreAuthorize("hasAuthority('USUARIO')")` direto.

## Consequências

### Positivas

- **Privilégio de emissão limitado:** comprometer o
  `restaurante-pedido` não dá ao atacante poder de emitir tokens —
  ele só possui a chave pública.
- **Roda em qualquer serviço novo:** basta colocar o `publicKey.pem`
  no classpath e configurar o `NimbusJwtDecoder`.
- **Padrão da indústria:** alinhado com OAuth2/OIDC, que usam RS256
  como default.

### Negativas

- **Chave pública precisa ser distribuída:** hoje vive como arquivo
  no classpath de cada serviço. Em uma evolução real, seria servida
  via endpoint JWKS dinâmico (`/.well-known/jwks.json`) que os
  serviços consultam na inicialização.
- **Performance:** RSA é mais lento que HMAC (~10× para assinar,
  ~100× para verificar). Para o volume do projeto, irrelevante.
- **Tamanho do token:** assinatura RSA-2048 é ~256 bytes vs ~32
  bytes de HMAC-SHA256. Cabeçalho HTTP fica maior, mas insignificante.

## Alternativas consideradas

- **HS256 com chave compartilhada:** mais simples, mais rápido,
  mas com risco de blast radius — qualquer validador comprometido
  pode emitir.
- **Auth-service centralizado validando todos os tokens
  remotamente (introspection):** cada serviço chamaria o auth-service
  para validar o token. Adiciona latência e cria dependência síncrona
  forte. Padrão OAuth2 introspection.
- **Sessões server-side (sticky cookies):** descartado por não ser
  stateless — exige sticky sessions ou um cache distribuído
  (Redis), e a spec exige stateless JWT.
